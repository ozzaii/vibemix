<objective>
Ship the **binary-attack verification gate** that VERIFY-04 has been pointing
at since the project charter. One Python script (`scripts/dist/verify_binary.py`)
walks any shipped vibemix bundle — `.app` on macOS or `.msi` on Windows —
extracts every PyInstaller archive it finds (via a vendored copy of the public-domain
`pyinstxtractor`), runs raw-byte scans against the canonical Google + AWS + OAuth + OpenAI
key patterns, and exits non-zero if anything matches. This is the load-bearing CI
gate that prevents Phase 18's release workflow (Plan 18-05) from publishing a
binary with a leaked `AIza...` key. Pure stdlib — no new pip dep.

Purpose: Close the binary-attack vector. Phase 11 W1 already proved the
packaging-time variant (0/482 files clean on Kaan's M-series build); Plan 18-01
generalises it to run against any post-codesign / post-Inno-Setup artifact in CI.

Output:
- `scripts/dist/verify_binary.py` (CLI + importable library)
- `scripts/dist/_pyinstxtractor.py` (vendored, attributed)
- `tests/dist/test_verify_binary.py` + fixtures + conftest
- Zero new pip / npm dependencies.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| post-codesign artifact → CI verifier | Untrusted build output (could contain leaked keys from a regression in build_sidecar's leak gate or a third-party dep ship). The verifier is what catches it. |
| vendored _pyinstxtractor.py → import path | Public-domain upstream code; the only attack surface is "someone in CI replaces the vendored file with a malicious one to skip the AIza scan". Mitigated by committing the file + standard PR review. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-01 | Information Disclosure | verify-report.json | mitigate | Report writer explicitly stores only pattern name + bundle-relative source path; never the matched byte sequence. Asserted by `test_report_redacts_matched_bytes`. |
| T-18-02 | Information Disclosure | log output during scan | mitigate | All log/print calls in verify_binary.py use pattern name + redacted bytes; `test_log_output_does_not_leak_planted_key` enforces with caplog. |
| T-18-03 | Tampering | vendored _pyinstxtractor.py | accept | Standard PR review + Apache-2.0 / Unlicense compatibility check. Replacing it with a malicious variant would still be caught by Plan 18-05 CI because the AIza scan would either trip on the planted-key test (test suite must still pass) or the malicious extractor would have to selectively suppress a single scan — out of scope for v1 threat model. |
| T-18-04 | Denial of Service | scan against an enormous .pyz | accept | Synthetic .pyz fixtures are kept tiny; in CI the artifact size is bounded by what build_sidecar produces (~242 MB total bundle on macOS). Scan completes in seconds. |
</threat_model>
