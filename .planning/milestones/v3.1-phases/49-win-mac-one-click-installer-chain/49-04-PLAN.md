<objective>
Wire the installer-level + Tauri-level glue that fires the companion scripts at the right moment per OS:
- **Mac**: bundled in DMG; `firstrun_companion.sh` invokes companion fetch on first launch (legal blocker = BlackHole .pkg redistribution; documented per ARCHITECTURE.md)
- **Win**: bundled in MSI/EXE; Inno Setup `[Run]` invokes companion fetch right after main install; `[Code]` dialog shows VB-CABLE license ack BEFORE fetch fires

Confirms Tauri MSI target (already present; verify) and exposes a `run_companion_fetch` Tauri command for wizard UI orchestration. Closes INSTALL-02 (Win flow) + INSTALL-06 (MSI + stopwatch glue).

Out of scope: companion script bodies (Plan 01), signing chain (Plan 02), wizard UI (Plan 03), VM matrix (Plan 05), anti-slop + uninstall (Plan 06).
</objective>

<interfaces>
Inno Setup `[Run]` runs AFTER all files are extracted. `[Code]` runs custom Pascal logic at various install lifecycle points (`InitializeWizard`, `NextButtonClick`, etc.). VB-CABLE license dialog runs at `InitializeWizard` so the user sees it before any fetch fires.

Mac DMG cannot legally bundle BlackHole `.pkg` (ExistentialAudio license). Companion fetch is deferred to first launch via `installer/macos/firstrun_companion.sh` which runs from the app's `Info.plist` `LSApplicationCategoryType` first-launch hook (or via a sentinel-file pattern in `~/Library/Application Support/vibemix/firstlaunch.done`).

Tauri shell capability scope must explicitly allow the companion script paths. Existing `tauri/src-tauri/capabilities/default.json` already scopes `shell:allow-execute` for sidecar binaries; this plan extends with the new `installer/companion/*` paths but adds ZERO new permission identifier (CONTEXT Decision 7 IPC invariant).

Step-0 worktree-sync invariant per memory `feedback_worktree_must_sync_main_first`.
</interfaces>
