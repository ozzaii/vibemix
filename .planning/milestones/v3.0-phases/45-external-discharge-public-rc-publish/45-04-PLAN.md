<interfaces>
The 4 evidence sources the audit reads:

Source 1 — GH releases telemetry:
- Read via `gh CLI` when --live; from `fixtures/synthetic_telemetry/gh_releases.json` otherwise.
- Fields needed: `download_count` (aggregate across all DMG + MSI assets), `published_at` (used to set the 14-day window anchor).
- Live shape: `gh api repos/bravoh/vibemix/releases --jq '[.[] | select(.tag_name=="v3.0.0-rc1")][0]'`.

Source 2 — Bravoh healthz cron uptime:
- Read from `fixtures/synthetic_telemetry/healthz_uptime.csv` when not --live.
- When --live: parse Bravoh server logs OR call a `/vibemix/healthz/stats` endpoint (Plan 45-06 §SHIP-06 runbook documents which).
- Fields needed: `total_checks` (= 14d × 24h × 12/h = 4032 max), `ok_count`, `uptime_pct = ok_count / total_checks * 100`. Target ≥99.5%.

Source 3 — Ear-test logs (Plan 42-03):
- Read all `eval/ear-test-logs/*.json` since `published_at`; schema per `eval/ear-test-logs/schema.json`.
- Count: total ear-test sessions, sessions flagged `felt_slop` / `felt_scripted` / `felt_late` / `felt_generic`, per-genre coverage.

Source 4 — GH issues labelled `crash` open since `published_at`:
- `gh issue list --label crash --state all --since YYYY-MM-DD --json number,title,createdAt,closedAt,state`
- Severity rollup: open vs closed, oldest-open age, count.

Aggregate output is written to `.planning/decisions/v3.0-SHIP-V1-DECISION.md` using `docs/SHIP-V1-DECISION-TEMPLATE.md` as the structural shell.
</interfaces>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| audit script → GH API | Read-only; never POST or PATCH or DELETE; GITHUB_TOKEN scope should be read-only-public (no admin scope required). |
| audit script → Bravoh healthz endpoint | Read-only HTTP GET; same threat envelope as Plan 45-03 probe. |
| audit script → ear-test logs (filesystem) | Local read; logs are project-private per Plan 42-06 privacy contract — audit MUST NOT leak log content into the decision report (only aggregate counts). |
| audit script → decision report (filesystem) | Write to .planning/decisions/; atomic write to prevent partial state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-45-04-01 | Tampering | --live response from GH API contains injected markdown | mitigate | All string interpolation into the template uses repr-safe escaping (json.dumps for numerics, no raw HTML); test asserts template output round-trips parse-safe. |
| T-45-04-02 | Info disclosure | audit accidentally writes raw ear-test log content into the decision report | mitigate | Plan 42-06 privacy contract preserved — audit only emits aggregate counts, never log content. Test 14 + a privacy-mirror test pins this. |
| T-45-04-03 | Repudiation | "did the audit actually run, or did Kaan hand-fill?" | mitigate | Audit writes a `_generated_by: audit_ship_v1_decision.py vNN at ISO-8601` HTML comment in the report; Kaan's manual edits are below the dividing line; runbook documents this. |
| T-45-04-04 | DoS | --live takes >5min due to GH rate limit | accept | Audit is a T+30 one-shot; rate limit non-issue at that cadence. |
| T-45-04-05 | Elevation of privilege | --live could mutate GH state via mis-typed flag | mitigate | argparse rejects flags that look like mutation; subprocess.run argv list passes literal `gh api ... --jq ...` — no `--method POST/PATCH/DELETE`; test pins this via subprocess argv inspection. |
