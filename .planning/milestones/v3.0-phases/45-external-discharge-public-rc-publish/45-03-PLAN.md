Ship the engineering side of SHIP-06 (OPS-14-SERVER): a 3-endpoint health probe that gates `cut_release.sh` from publishing until the Bravoh team has deployed the updater endpoints + cron heartbeat. <probe_contract>
<!-- The 3 contractual endpoints + their expected behaviour. -->

Default endpoint base: `https://api.altidus.world` (Bravoh proxy host).
Override via `--endpoint-base URL` (for staging or fixture mock).

1. **GET /vibemix/healthz** — must return HTTP 200 with JSON body `{"status":"ok","ts":"<ISO-8601>"}`; `ts` ≤ 10 min old per CONTEXT §SHIP-06 cron `*/5 * * * *` requirement.
2. **GET /vibemix/updates/latest.json** — must return HTTP 200 with JSON body `{"version": "<semver>", ...}` (any well-formed updater manifest).
3. **POST /vibemix/updates/upload (HEAD only in probe)** — must return HTTP 401 (auth required) or 200 — anything in {200, 401, 405} is "endpoint exists"; 404 = missing. The probe does NOT POST a real upload; it sends `HEAD` to detect endpoint mount + accepts 401/405 as "endpoint present + auth-gated correctly".

Exit codes:
- 0 = all 3 endpoints OK + healthz fresh
- 1 = at least one endpoint missing (404)
- 2 = CLI usage error (bad flag, missing curl/jq)
- 3 = network failure (DNS / TCP / TLS) on any endpoint
- 4 = healthz reachable but stale (ts older than 10 min, signals cron not running)
</probe_contract>
</context>
