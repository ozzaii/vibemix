<objective>
Port `cohost_v4_tr.py:624-770` `LookaheadProvider` verbatim into `src/vibemix/audio/lookahead.py`. The provider peeks 3s ahead into the source file currently playing in djay Pro by chaining `nowplaying-cli` (get title + elapsed time) → `mdfind` (find file path via Spotlight) → `ffmpeg` (extract 18s window ending 3s past playhead). All failure paths return `(None, meta)` — the caller in Plan 40-03 branches on `if lookahead_wav:` and never on exceptions.

**Hallucination class closed:** "AI reacts after the moment passed" — Gemini sees ~3s of audio the audience hasn't heard yet, so the LLM+TTS latency (typically 700-1500ms) is offset by the lookahead window. The AI's reaction "lands timed to the moment it describes" instead of trailing it by a perceptible gap.

Purpose: Anti-slop temporal grounding — convert past-only audio context into past+near-future context for the LLM, masking pipeline latency.

`subprocess.run` invocation pattern (RESEARCH §V10 Malicious Code): ALWAYS list-form, NEVER `shell=True`. Track titles from `nowplaying-cli` are untrusted external input — pass them as plain argv elements to `mdfind`, never interpolate into a shell command string.

Brew-installed binaries (verified present per RESEARCH §Environment Availability):
- `/opt/homebrew/bin/nowplaying-cli` — Returns JSON-ish records via `nowplaying-cli get-raw`. Falls back to `shutil.which("nowplaying-cli")` if the brew path is absent (e.g. Linux dev hosts the test suite runs on — should gracefully return None there).
- `/usr/bin/mdfind` — macOS stdlib (always present on macOS). Use `mdfind -name "<title>"` to find local audio files.
- `/opt/homebrew/bin/ffmpeg` — Required for the extract step. RESEARCH Pattern 3 confirms `-ss` BEFORE `-i` is mandatory for fast seek.

Constants to add to `lookahead.py` (module-scope, NOT into `audio/constants.py` — these live with the provider per "Claude's Discretion" in CONTEXT):
- `LOOKAHEAD_SECONDS: float = 3.0` — Window END this far ahead (LLM+TTS latency offset).
- `LOOKAHEAD_WINDOW_SECONDS: float = 18.0` — Total window length (15s past context + 3s future).
- `LOOKAHEAD_SAMPLE_RATE: int = INPUT_SR_TARGET` — Matches Part 1 / Part 2 rate (16kHz).
- `LOOKAHEAD_TIMEOUT_S: float = 4.0` — ffmpeg subprocess wall-clock ceiling.

ffmpeg invocation (verbatim from cohost_v4_tr.py:747-755 per RESEARCH Pattern 3):
```
[ffmpeg, "-loglevel", "error",
 "-ss", f"{seek:.3f}",   # BEFORE -i: fast input seek
 "-i", path,
 "-t", f"{duration:.3f}",
 "-ac", "1",             # mono
 "-ar", "16000",         # 16kHz
 "-f", "wav",            # WAV container
 "-y", "pipe:1"]
```
Note: `-f wav` (not `-f s16le`) per RESEARCH Open Question 3 recommendation — matches existing `snapshot_wav` format used everywhere else in `dj_cohost.py`.

`snapshot_wav` return shape: `tuple[bytes | None, dict]` where:
- `bytes` = 18s mono 16kHz WAV payload on success, `None` on every failure path.
- `dict` always contains `"ok": bool` and `"reason": str` (e.g. "no nowplaying", "no file", "ffmpeg error: <stderr>", "timeout", "exception: <e>"). Optionally contains `"title": str`, `"position_s": float`, `"path": str`, `"wall_ms": float` for diagnostics.

`_current_position` v4_tr extrapolation guard (cohost_v4_tr.py:717-719): When `prev_title == title` and recent raw poll, extrapolate `elapsedTime` by adding `now - last_raw_wall` × `playbackRate`. When title changes, do NOT extrapolate — return the fresh raw value. This is the Pitfall 2 (nowplaying-cli stale data on app switch) defense.

Title→path cache (`_title_to_path: dict[str, str | None]`): Per-instance dict that caches mdfind results. Stores `None` for titles that returned no Spotlight hit (so we don't re-query on every snapshot for streaming-only tracks). RESEARCH Open Question 3 recommends per-session lifetime — the cache resets when `DJCoHostAgent` is re-instantiated.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| nowplaying-cli output → LookaheadProvider | Untrusted external JSON; track title is user-controlled (any app can publish via MediaRemote) |
| mdfind output → LookaheadProvider | Spotlight-indexed file paths; trusted by macOS but path content reflects user filesystem |
| ffmpeg input file → LookaheadProvider stdout | Arbitrary audio file content; trusted by ffmpeg's own input validation (we did not write the decoder) |
| LookaheadProvider stdout → Gemini Part 3 (Plan 40-03) | Mic + lookahead audio crosses to genai API — privacy boundary same as existing Part 1 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-40-02-01 | Tampering | mdfind track-title interpolation | mitigate | All subprocess calls use list-form (`subprocess.run([bin, "-name", title])`) — title is plain argv, never shell-expanded. Test `test_subprocess_args_use_list_form` pins this. |
| T-40-02-02 | DoS | ffmpeg on malformed input | mitigate | 4-second `timeout=LOOKAHEAD_TIMEOUT_S` on every ffmpeg invocation; `subprocess.TimeoutExpired` caught and returned as `(None, meta)`. Test `test_subprocess_timeout_returns_none` pins this. |
| T-40-02-03 | Tampering | ffmpeg moov-atom-not-found | mitigate | Pitfall 4 — non-zero ffmpeg returncode with stderr containing "moov atom not found" returns `(None, meta)` with the reason logged; never raises. Test `test_ffmpeg_error_returns_none` pins this. |
| T-40-02-04 | Spoofing | nowplaying-cli responding to other-app spoof | mitigate | Read only the 3 fields we trust (`title`, `elapsedTime`, `playbackRate`); validate `elapsedTime` is a non-negative float; never `eval()`/`exec()` the JSON. Pitfall 2 extrapolation guard catches sustained mismatches by NOT extrapolating across title changes. Test `test_extrapolation_guard_on_title_change` pins this. |
| T-40-02-05 | Information Disclosure | track file paths in meta dict | accept | `meta["path"]` populated only in diagnostic mode and never logged outside the per-session `events.jsonl` (which is gitignored). Privacy posture matches existing `events.jsonl` per CLAUDE.md `feedback_privacy_scope_narrow`. |
</threat_model>
