<task id="1" type="auto">
<name>proxy_load_test.py — 100 RPS × 5min smoke against Bravoh proxy</name>
<behavior>
A Python script at `scripts/dayzero/proxy_load_test.py` that fires N concurrent HTTPS requests/second at a target URL for M seconds and reports min/median/p95/p99 latency, error rate, and a pass/fail verdict against p99<500ms + error_rate<1% gates.

Defaults: target=https://api.altidus.world/vibemix/healthz, rps=100, duration=300s, concurrency=20. CLI overridable.

`--dry-run` short-circuits actual HTTP: the script generates synthetic latency samples (gaussian around 200ms, occasional outliers) and runs the full report path. This lets us test the gating logic without hitting Bravoh's staging.

Stdlib + httpx only. No locust dep, no asyncio framework lock-in beyond what httpx already brings. Reuses Python 3.14 + .venv.
</behavior>
<implementation>
1. Create `scripts/dayzero/__init__.py` (empty, just for package shape).
2. Create `scripts/dayzero/proxy_load_test.py`:
   - argparse: `--target` (URL, default `https://api.altidus.world/vibemix/healthz`), `--rps` (int, default 100), `--duration` (seconds, default 300), `--concurrency` (int, default 20), `--p99-budget-ms` (default 500), `--error-rate-budget` (default 0.01), `--dry-run` (flag), `--json` (machine-readable output flag).
   - Uses `httpx.AsyncClient` with HTTP/2 enabled, gather waves of requests sized to hit target RPS.
   - Records (sample_idx, t_request_ms, t_response_ms, status, latency_ms) per sample.
   - At end: computes min/median/p95/p99/max latency, success/error rates, prints a table.
   - Verdict: PASS if p99 < p99-budget-ms AND error_rate < error-rate-budget, else FAIL.
   - Exit code: 0 on PASS, 1 on FAIL.
   - `--dry-run` mode: skips httpx entirely, generates synthetic samples via `random.gauss(0.2, 0.05)` with 1% chance of synthetic-failure status (599) and 0.5% chance of synthetic-outlier latency (random.uniform(0.6, 1.2)s). Verdict logic still runs.
   - `--json` mode: machine-readable JSON to stdout (final summary), human-readable still goes to stderr.
3. Use stdlib `time.monotonic()` for timing; stdlib `statistics` for percentile-adjacent stats (use a manual sorted-percentile since `statistics.quantiles` is awkward for p99).
4. Conservative concurrency model: a `Semaphore` caps in-flight, a simple sleep-throttle keeps RPS at target. No fancy token bucket.
</implementation>

<task id="2" type="auto">
<name>healthz_check.sh — curl loop with non-200 alert</name>
<behavior>
A shell script at `scripts/dayzero/healthz_check.sh` that polls a healthz endpoint every N seconds (default 30) and emits a structured alert line to stderr on any non-200 response. Foreground process by design — Kaan tails it in a terminal during launch.

`--dry-run` mode runs the alert/logging path against a synthetic injection schedule (e.g. simulate 1 non-200 every 90 seconds) so the script can be tested without hitting a real endpoint.
</behavior>
<implementation>
1. Create `scripts/dayzero/healthz_check.sh` (bash, `set -euo pipefail`).
2. Args: `--target URL` (default `https://api.altidus.world/healthz`), `--interval SECONDS` (default 30), `--max-iterations N` (default 0 = infinite, used by tests), `--dry-run` (synthetic), `--alert-cmd CMD` (optional shell command to invoke on non-200; default = stderr line only).
3. Each iteration:
   - Capture `t_start_iso=$(date -u +"%Y-%m-%dT%H:%M:%SZ")`.
   - If `--dry-run`: synthesize a status code — return 503 on every 3rd iteration, 200 otherwise.
   - Else: `curl -s -o /dev/null -w "%{http_code}" --max-time 10 "$TARGET"`.
   - If status != 200: emit `[ALERT] iso=... target=... status=... iteration=...` to stderr, run `--alert-cmd` if set.
   - Else: emit `[OK] iso=... target=... status=200 iteration=...` to stdout.
   - Sleep `--interval` seconds.
4. Handle SIGINT / SIGTERM cleanly — print a summary line `[SUMMARY] iterations=N ok=N alerts=N` and exit 0.
5. `--max-iterations` allows tests to run a bounded sweep without modifying timing.
6. Make executable: `chmod +x scripts/dayzero/healthz_check.sh`.
</implementation>
