Kaan locally** can take a built `.app` and produce a Gatekeeper-accepted,
notarized, stapled DMG in one command. The three artifacts:

1. `tauri/src-tauri/entitlements.macos.plist` — the canonical Hardened Runtime

   notarytool → staple → spctl-verify, with retries on transient notarytool
   failures.

This is the scaffolding-only phase: no actual signing runs here (that requires
Kaan's Apple Developer ID identity + ASC API key). Plan 18-05 wires the script
into CI; Phase 20 owns the fresh-machine rehearsal.

Purpose: Make signing reproducible. The wedge of a polished OSS launch is that
"download → drag to Applications → it just opens" works on the first try. The
sign script is what closes the gap between PyInstaller output and Gatekeeper
acceptance.

        codesign --sign "$APPLE_DEVELOPER_ID" --force --options runtime --entitlements "$ENTITLEMENTS" --timestamp "$file"

      Filter the find list to skip already-signed files where `codesign --verify` succeeds (idempotency).

    Stage 3 — Codesign the .app bundle with --deep + --options runtime:
        codesign --sign "$APPLE_DEVELOPER_ID" --force --deep --options runtime \
                 --entitlements "$ENTITLEMENTS" --timestamp "$APP"

      Then verify strict:
        codesign --verify --deep --strict --verbose=2 "$APP"

    Stage 4 — Create DMG (skip if --skip-dmg):
        create-dmg \
          --volname "vibemix" \
          --volicon "$REPO_ROOT/tauri/src-tauri/icons/icon.png" \
          --window-pos 200 120 --window-size 600 400 \
          --icon-size 100 \
          --app-drop-link 425 200 \
          "$DMG_OUT" "$APP"

      DMG name: `vibemix-$(plutil -extract CFBundleShortVersionString xml1 -o - "$APP/Contents/Info.plist" | grep -oE '[0-9]+\.[0-9]+\.[0-9]+').dmg`. If version extraction fails, fall back to `vibemix.dmg` with a stderr WARNING (don't fail the run).

    Stage 5 — Submit DMG to notarytool with --wait (10-minute polling timeout); idempotent retry on transient HTTP / network errors:
        for attempt in 1 2 3; do
          xcrun notarytool submit "$DMG_OUT" \
            --key "$APPLE_API_KEY_PATH" \
            --key-id "$APPLE_API_KEY_ID" \
            --issuer "$APPLE_API_KEY_ISSUER" \
            --wait \
            && break
          echo "[sign_macos] notarytool attempt $attempt failed; retrying in 30s" >&2
          sleep 30
        done

      After success, fetch the notarization log via `xcrun notarytool log <submission-id>` and print the JSON.

      If all 3 attempts fail, exit 3 with the log JSON dumped to stderr.

    Stage 6 — Staple:
        xcrun stapler staple "$DMG_OUT"

      Verify:
        xcrun stapler validate "$DMG_OUT"

    Stage 7 — Gatekeeper assess (final gate):
        spctl --assess --type execute --verbose "$APP"

      Expect output containing `source=Notarized Developer ID`. If it reports `accepted` but with a different source, print a WARNING but exit 0 — the signed DMG is shippable. If `rejected`, exit 4.

    Stage 8 — Run verify_binary.py against the signed DMG (cross-plan integration with Plan 18-01):
        uv run python -m scripts.dist.verify_binary "$APP" --report "$OUTPUT_DIR/verify-report.json"

      If verify_binary exits non-zero, exit 5 (release-blocking — even a notarized binary cannot ship if it contains an AIza string).

    Idempotency: Re-running on an already-signed + notarized + stapled .app should be a no-op (Stage 2/3 detect existing signatures via `codesign --verify`; Stage 6 detects existing stapler ticket via `stapler validate`).

    Logging: every stage prints `[sign_macos] stage N: <description>` to stderr; success exits 0; failure exits 1 from `set -e` or an explicit numeric exit code as above.
