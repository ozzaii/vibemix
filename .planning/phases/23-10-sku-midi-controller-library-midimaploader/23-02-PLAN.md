<objective>
Ship the 10-SKU controller library + MidiMapLoader. Schema-validated JSON
per controller replaces the hardcoded `_CC_MAP` / `_NOTE_MAP` dicts in
the POC. MidiMapLoader is the universal grounding spine: every DJ app
with a supported controller surfaces normalized semantic events
(eq_low_a, fader_b, sync_a, cue_b, jog_a, ...) regardless of vendor /
model.

10 SKUs (per orchestrator spec for v2.0):
- Pioneer DDJ-FLX4 (POC baseline, sync note pending FLX4-SNIFF.md verdict)
- Pioneer DDJ-400, DDJ-200, DDJ-REV1
- Denon MC7000, MC6000
- Native Instruments Kontrol S4 Mk3, Kontrol S2 Mk3
- Numark Mixtrack Platinum FX, Mixtrack Pro FX

For non-FLX4 SKUs where exact CC/note bindings can't be grounded in
research, the JSON marks the entry `status: "tentative — needs hardware verification"`
so the registry never lies (Pitfall 24 — telemetry surfaces honest flag).
</objective>

JSON schema (Draft-07):
{
  "vendor": string,             # "Pioneer" / "Denon" / "Native Instruments" / "Numark"
  "model": string,              # "DDJ-FLX4" / "DDJ-400" / ...
  "description": string,        # one-line human description
  "verified": bool,             # true if hardware-sniffed, false if research-grounded
  "controls": {
    "<semantic_name>": {
      "type": "cc" | "note",
      "channel": 0-15,
      "value": 0-127,            # cc number or note number
      "semantic": string,        # e.g. "eq_low_a", "fader_b", "sync_a"
      "status": "verified" | "tentative — needs hardware verification" | "pending-verdict"
    },
    ...
  }
}

Semantic vocabulary (matches cohost_v4 + Plan 23-CONTEXT):
- EQ:        eq_low_a, eq_low_b, eq_mid_a, eq_mid_b, eq_hi_a, eq_hi_b
- Volume:    vol_a, vol_b
- Tempo:     tempo_a, tempo_b
- Filter:    filter_a, filter_b
- Crossfader: xfader
- Buttons:   play_a, play_b, cue_a, cue_b, sync_a, sync_b, jog_a, jog_b,
             loop_in_a, loop_in_b, loop_out_a, loop_out_b
- Sync alt (defensive, FLX4 only until verdict): sync_a_alt, sync_b_alt

MidiMapLoader public API:
  ml = MidiMapLoader()                      # auto-discovers controllers/*.json
  cmap = ml.load("ddj-flx4")                # returns dict with vendor/model/controls
  all_maps = ml.all_maps()                  # dict[id -> map]
  semantic = ml.lookup(cmap, msg)           # msg is mido.Message; returns "eq_low_a" or None

mido.Message attrs used by lookup (no mido import required at load time):
- msg.type in {"control_change", "note_on", "note_off"}
- msg.channel: int 0-15
- msg.control: int (when type == control_change)
- msg.note:    int (when type == note_on / note_off)

  <name>Task 2: MidiMapLoader class + tests</name>
  <files>src/vibemix/midi/map_loader.py, tests/midi/test_map_loader.py</files>
  <behavior>
- `MidiMapLoader()` constructor: auto-discovers JSONs from `src/vibemix/midi/controllers/`. Validates each against `schema.json` using `jsonschema.validate`. Raises `MapValidationError` on any file failing schema. Map ID derived from filename stem (e.g. `ddj-flx4.json` → `"ddj-flx4"`).
- `.load(controller_id: str) -> dict` returns the validated map dict; raises `KeyError` with helpful message listing available IDs if not found.
- `.all_maps() -> dict[str, dict]` returns the full registry.
- `.lookup(controller_map: dict, msg) -> str | None` — for a mido Message-like object (duck-typed: needs `.type`, `.channel`, plus `.control`+`.value` OR `.note`+`.velocity`), returns the matching semantic name or None. Note_off resolves the same as note_on (semantic is button-press, not press-vs-release). MidiMapLoader is read-only after construction (no mutation API in v2.0).
  </behavior>
  <implementation>
- File-level docstring + Apache-2.0 license header matches repo style.
- Module-level constants: `CONTROLLERS_DIR = Path(__file__).parent / "controllers"`, `SCHEMA_PATH = Path(__file__).parent / "schema.json"`.
- `MapValidationError(Exception)` raised on schema failures; message includes the filename and the underlying jsonschema error.
- Lookup builds an index dict on map-load for O(1) lookup: `{(type, channel, value): semantic}`. Note_on and note_off share the same lookup keys (semantic event = press event).
- Lookup returns None for messages not in the index (unmapped CC/note, or msg.type not in {control_change, note_on, note_off}).
- Tests:

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Disk → process | Controller JSONs are loaded from the source tree at runtime; schema validation rejects malformed maps before they reach the MIDI ingest layer. |
| MIDI input → MidiMapLoader.lookup | Untrusted USB MIDI device sends arbitrary bytes; lookup is a pure dict-index O(1) operation and cannot raise on malformed input — returns None for anything not in the registry. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation |
|-----------|----------|-----------|-------------|------------|
| T-23-04 | Tampering | controllers/*.json | mitigate | Schema validation on load (jsonschema Draft-07); a tampered JSON shipped to disk that violates the schema fails MidiMapLoader construction with MapValidationError citing the file. test_loader_invalid_json_raises_map_validation_error pins this behavior. |
| T-23-05 | Information Disclosure | MidiMapLoader registry | accept | All loaded maps are public reference data (vendor + model + CC/note bindings); no PII, no credentials, no env. |
| T-23-06 | Spoofing | MIDI device with cloned port name | accept | If a malicious USB device clones the DDJ-FLX4 port name, MidiMapLoader.lookup will produce semantic names for whatever notes the device emits — but the worst case is mismapped DJ events, not privilege escalation. Acceptable in v2.0 (Phase 26 hardening reconsiders). |
</threat_model>
