    **# vibemix proxy** — short tagline: "FastAPI gateway keeping the Gemini API key off vibemix's distributed binaries."

    **## What it does** (2 paragraphs)
    - Receives bearer-JWT-authenticated requests from vibemix clients.
    - Pass-through proxies LLM (`/v1beta/models/{model}:streamGenerateContent` + sibling non-streaming) and TTS (`/v1/audio/speech`) calls to upstream Gemini + OpenRouter with the proxy's own keys.
    - Issues install-UUID JWTs via `/api/vibemix/v1/register` (idempotent).
    - Per-UUID rate limit (60 rpm) + Redis-backed daily quota (2000 rpd). IP-keyed limit on `/register`.
    - Circuit breaker on consecutive upstream 5xx → 503 + Retry-After.

    **## Architecture** (1 paragraph + diagram)
    Embed the ASCII architecture diagram from RESEARCH Q-Architecture-Diagram. Note that the proxy is independent of Bravoh's main app (separate service alongside bravoh-api / bravoh-celery on the same server). Both share Redis.

    - Container logs: `docker logs -f vibemix-proxy`
    - PM2 logs: `pm2 logs vibemix-proxy`

    **## JWT_SECRET rotation**
    Rotating JWT_SECRET invalidates ALL outstanding JWTs (90-day TTL × every active install). Procedure:
    1. Generate new secret: `python3 -c "import secrets; print(secrets.token_urlsafe(64))"`
    2. Update `proxy/.env` on server: `JWT_SECRET=<new>`
    3. Restart proxy: `docker restart vibemix-proxy` (or `pm2 restart vibemix-proxy`)
    4. Existing clients will hit 401 on next request → vibemix client's `get_or_refresh_jwt` detects this and re-calls `/register` automatically. Recovery is transparent to the user.

    Rotate ONLY on suspected compromise (e.g., server breach, leaked .env). The 90-day TTL + per-UUID rate-limit cap damage from any single leaked JWT.

    **## Troubleshooting**

    | Symptom | Likely cause | Fix |
    |---------|--------------|-----|
    | 404 on `/v1beta/models/.../streamGenerateContent` | nginx location block missing | Add the `/v1beta/` location block above |
    | SSE chunks arrive batched (no streaming feel) | nginx `proxy_buffering on` (default) | Set `proxy_buffering off` in the location block (Pitfall 2) |
    | Daily quota counter never expires | Redis < 7.0 (Pitfall 4) | `redis-cli INFO server` → if `redis_version < 7.0`, upgrade Redis OR use the LUA fallback in RESEARCH Q5 |
    | Every vibemix launch hits `/register` | Client keyring is null backend (Pitfall 6) | Client should fall back to file at `~/Library/Application Support/vibemix/install_uuid` — check that file exists and contains 32-hex-char value |
    | 401 with `{"detail":"invalid token"}` after JWT rotation | Expected — clients auto-recover on next launch via `get_or_refresh_jwt` | No action; document in user-facing release notes that "JWT rotation requires app relaunch" |
    | Circuit breaker stuck open | 10 consecutive upstream Gemini/OpenRouter 5xx | Check upstream status pages; breaker auto-resets after 60s |
    | Container won't start | Missing required env var in `.env` | Check stderr: pydantic ValidationError lists the missing field name (never the value — safe to log) |

    **## Security notes**
    - All keys (`GEMINI_API_KEY`, `OPENROUTER_API_KEY`, `JWT_SECRET`) live ONLY in `proxy/.env` on the server. Permissions: `chmod 600 proxy/.env`. Never commit.

    - Container runs as non-root (uid 10001 `appuser`).
    - Vibemix clients (PyInstaller binaries) contain ZERO upstream-API-key strings. CI regression test pins this.
    - HTTPS is terminated at nginx; the proxy container speaks plain HTTP on 127.0.0.1:8788 only.
