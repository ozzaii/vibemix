## Decisions Made

- **Spec file naming convention is `<name>.<platform>.spec`** — NOT the plan's suggested `<name>.spec.<platform>`. PyInstaller 6.x switches into "spec mode" ONLY when the filename has a literal `.spec` suffix. With the original naming PyInstaller fell into "script mode" and ignored the `COLLECT(name='vibemix-core')` argument, producing `dist/vibemix.spec/vibemix.spec` instead of `dist/vibemix-core/vibemix-core`. Renaming is the canonical fix; documented in the file headers.
- **AIza scan in raw-bytes mode** — `subprocess.run(text=True)` crashes on Mach-O `.so` files containing non-UTF8 bytes (Pillow's `.so` at position 2713 was the first repro). AIza is pure ASCII so the byte regex matches the raw output directly; ASCII-with-errors-replace decode happens ONLY for the human-readable error log line.
- **Bundle ID `world.bravoh.vibemix` locked in entitlements.plist NOW** — Phase 18 owns codesigning but A8 establishes that macOS TCC permissions are keyed to the bundle ID. Lock-and-document discipline means any future change invalidates every user's granted permissions. Pinned in the entitlements file with a load-bearing comment.
- **Entitlements scope: minimum viable** — `audio-input` + `microphone` are product-essential; `allow-unsigned-executable-memory` is required for PyInstaller's bootloader + C-ext memory pages; `disable-library-validation` is required for LiveKit / Gemini / sounddevice `.dylib` loading (signed upstream, not by Bravoh). Deliberately NOT added: `allow-jit` (no JIT), `allow-arbitrary-loads` (network allowlist enforced by Tauri capability config in Wave 2), `camera` / `address-book` / `location` (zero use cases), `get-task-allow` (we don't debug shipped builds).
- **PyInstaller as dev-only dep** — added to `[dependency-groups] dev`, NOT `[project] dependencies`. End-user installs from PyPI (Phase 18+) don't pull PyInstaller's 33 MB. Only the build host runs it.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Spec file extension misordered for PyInstaller's `.spec` detection**
- **Found during:** Manual-checkpoint full PyInstaller build on Kaan's rig.
- **Issue:** Plan named the specs `vibemix.spec.macos` / `vibemix.spec.windows`. PyInstaller 6.x's spec-mode detection checks for the literal `.spec` suffix; with `.macos` / `.windows` as the LAST extension, PyInstaller fell into script-mode and ignored the `COLLECT(name='vibemix-core')` argument. Output landed at `dist/vibemix.spec/vibemix.spec` instead of `dist/vibemix-core/vibemix-core`. `scripts/build_sidecar.py` raised RuntimeError finding the path.
- **Fix:** `git mv` to `vibemix-core.macos.spec` / `vibemix-core.windows.spec`. Updated docstrings in `scripts/build_sidecar.py` (3 places) + spec file header comments (2 places). The COLLECT `name=` is now honored; binary lands at the expected `tauri/src-tauri/binaries/vibemix-core-aarch64-apple-darwin/vibemix-core-aarch64-apple-darwin` path.
- **Files modified:** `vibemix-core.macos.spec`, `vibemix-core.windows.spec`, `scripts/build_sidecar.py`.
- **Verification:** Full PyInstaller build exits 0; binary path matches `scripts/build_sidecar.py`'s expected output.
- **Committed in:** `55d4c04` (fix).

**2. [Rule 1 — Bug] `_scan_file_for_aiza` UnicodeDecodeError on Mach-O binaries**
- **Found during:** Manual-checkpoint AIza gate run on the freshly built bundle.
- **Issue:** `subprocess.run(["strings", "-a", path], text=True)` decoded the `strings` output as UTF-8. Pillow's `.so` contains non-UTF8 bytes (0x97 at position 2713 of `_imaging.cpython-312-darwin.so`); UTF-8 decode raised `UnicodeDecodeError` and aborted the gate.
- **Fix:** Drop `text=True`. Match the AIza byte regex against raw `proc.stdout` bytes. AIza is pure ASCII so the byte pattern catches every leak without text decoding. Decode happens only for the redacted human-readable error log via `bytes.decode("ascii", errors="replace")`.
- **Files modified:** `scripts/build_sidecar.py` (`_scan_file_for_aiza` ~12 lines).
- **Verification:** `assert_no_aiza_leak(Path("dist/vibemix-core"))` exits 0, prints `"OK: no AIza-pattern strings found in bundle (482 file(s) scanned ...)"` to stderr. All 15 `test_build_sidecar_rename.py` tests still pass.
- **Committed in:** `55d4c04` (fix).

**3. [Rule 1 — Bug] `_FAKE_AIZA` test fixture too short**
- **Found during:** First run of `test_assert_no_aiza_leak_detects_literal_string`.
- **Issue:** Test fixture `_FAKE_AIZA = "AIza1234567890abcdefghijklmnopqrstuvwx"` had 34 characters after the `AIza` prefix; the regex pattern requires `{35}`. Test was a false-negative — `assert_no_aiza_leak` correctly returned None on the malformed sample. Caught immediately.
- **Fix:** Add one character: `"AIza1234567890abcdefghijklmnopqrstuvwxy"` (4 + 35 = 39 chars total). Regex now matches.
- **Files modified:** `tests/sidecar/test_build_sidecar_rename.py` (1 line).
- **Verification:** All 5 `assert_no_aiza_leak` tests pass.
- **Committed in:** `a4bbfbe` (in the Task 2 commit; bug was caught + fixed before commit).

**4. [Rule 1 — Bug] Stub stderr assertion text mismatch**
- **Found during:** First run of `test_wizard_stub_exits_cleanly`.
- **Issue:** Test asserted `"wizard mode not yet implemented"` but the stub prints `"[wizard] mode not yet implemented"`. The word "wizard" appears only once in the actual output (as part of the `[wizard]` prefix), not twice.
- **Fix:** Drop the literal "wizard" from the substring match. Now asserts `"mode not yet implemented" in proc.stderr` which catches the actual output without coupling to the prefix format.
- **Files modified:** `tests/sidecar/test_wizard_entrypoint.py` (1 line).
- **Verification:** All 7 `test_wizard_entrypoint.py` tests pass.
- **Committed in:** `c415a0c` (in the Task 1 commit; bug was caught + fixed before commit).
