# Architecture Research — v3.1 Distribution-Ready Pass

**Domain:** Subsequent-milestone integration into a 4-process shipped architecture (Tauri shell + Python sidecar + Bravoh proxy + mascot webview), adding distribution polish on top of the v3.0 "Clean OSS Ship" baseline.
**Researched:** 2026-05-17
**Confidence:** HIGH on existing-architecture surface (verified against `.planning/codebase/ARCHITECTURE.md`, `release.yml`, `install_vm_matrix.{sh,json}`, IPC schema, mascot module tree); MEDIUM on new-component placements (installer companion script + dep-audit surface are first-time integrations that ride existing seams); LOW on bundled-driver signing semantics (driver license + redistribution mechanics depend on KAAN-ACTION-LEGAL discharge timing).

---

## I. EXECUTIVE SUMMARY

v3.1 is a **distribution polish layer** — zero new product surfaces, zero new processes, zero new modules outside three existing roots (`installer/`, `scripts/release/`, `tauri/ui/assets/mascot/animations/`). The thesis:

1. **The release.yml matrix is the right surface for the one-click installer** — driver pre-flight runs as a NEW pre-package step BEFORE the existing sign/package/publish chain, and a NEW post-install companion (driver auto-fetch) ships as bundled data signed by the same SignPath/notarytool path that already covers the outer installer. No new CI job, no new pipeline.
2. **AUDIT.md is the dep-audit surface, lives at `docs/AUDIT.md`, and is generated by a new `scripts/audit/dep_audit.py`** that consumes existing `uv.lock` + `tauri/src-tauri/Cargo.lock` + `tauri/ui/package-lock.json` outputs. Hooks into the existing `python-cve.yml` + `rust-cve.yml` + `sbom.yml` workflows as a "dep-rating freshness" gate — does NOT replace osv-scanner / pip-audit / cargo-deny, augments them with a 3-color install-impact rating column.
3. **`tests/e2e/macbook/` is the right home for the new e2e harness** — sibling to existing `tests/e2e/test_seam_*.py` integration tests, but invokes the **built `.dmg`** through a new `scripts/e2e/run_macbook_pass.sh` driver instead of the in-process unit/integration regime. Visual diff artifacts land in `dist/e2e-macbook-runs/<UTC>/` (gitignored, same pattern as `dist/install-vm-runs/`).
4. **Real mascot GLBs land at `tauri/ui/assets/mascot/animations/prep_*.glb`** — slots already exist and bundle gate (`scripts/mascot/check_bundle_size.sh` Tier 2) is already failing red on placeholders. Real assets simply make Tier 2 turn green. 25 MB total bundle ceiling holds — v3.1 does NOT expand the ceiling; if a real retarget pushes over, the gate fails the PR and forces draco re-tuning.
5. **Dep-opportunity scan lives at `docs/dep-opportunities/<YYYY-MM>-scan.md`** — same pattern as `eval/THRESHOLD-RECALIBRATION-LOG.md` (dated artifact, append-only). New `scripts/audit/scan_opportunities.py` produces the scan; ADR-style decisions live at `.planning/decisions/DEP-OPP-<id>.md` (sibling to `P85-OVERRIDE-RETIRED.md`).

**The fifth invariant — "no new IPC messages"** — holds. Every v3.1 feature is either build-time (installer, dep-audit, GLB swap-in, dep-scan) or test-harness (e2e), so the runtime sidecar↔Tauri IPC schema stays frozen at v3.0's count. The 3-IPC-reservation contract (Phase 25 / debrief slot) is untouched. This is intentional architecture compression — v3.1 is a SHIP-quality milestone, not a feature milestone.

## III. v3.1 FEATURES — INTEGRATION INVENTORY

### Feature 1 — One-Click Installer (Win + Mac, auto-driver fetch + audio config)

**Goal:** Single download → run → ready-to-use, ZERO terminal commands. Closes the v3.0 INSTALL-VM-RUN gap that still expects users to `brew install blackhole-2ch` and `nowplaying-cli` themselves.

#### Architecture decision: bundled companion + post-install driver fetch (NOT bundled drivers)

The installer architecture has three reasonable shapes. Researched against v3.0 release.yml + signing constraints + driver license terms:

| Approach | Cost / Risk | Verdict |
|----------|-------------|---------|
| **(A) Bundle BlackHole + VB-CABLE driver binaries inside the DMG/MSI** | BlackHole is MIT-licensed (Existence Type) so technically OK to redistribute. VB-CABLE is freeware-with-attribution — redistribution requires explicit permission from VB-Audio. Adds ~12 MB to DMG (BlackHole) + ~3 MB (VB-CABLE) — within the 350 MB ceiling. SIGNING: the bundled `.pkg` (BlackHole) needs to be re-signed under Bravoh's Developer ID OR run the original signed installer via `installer -pkg`. macOS will refuse to run an unsigned re-bundled pkg post-Gatekeeper. **Recommendation: REJECT** — adds redistribution-license risk for VB-CABLE (memo from VB-Audio: "redistribution requires written agreement"), couples Bravoh's signing identity to third-party driver versioning, and bumps the DMG over a psychological 100 MB threshold | ❌ Rejected |
| **(B) Post-install companion script fetches drivers from official sources** | First-launch wizard step → companion script downloads BlackHole 0.6.x from existential.audio (signed pkg) + VB-CABLE from vb-audio.com (signed exe), runs each native installer with admin elevation, then probes. Adds ~3-8s to first launch (network dependent). License-clean (no redistribution). SIGNING: only the companion script is signed by Bravoh; driver pkgs are signed by their respective owners. **Recommendation: ADOPT** — license-clean, signing-clean, license-clean, keeps DMG under 100 MB, lets driver vendors handle their own update cycles | ✅ Selected |
| **(C) "Bring your own audio" — direct user to install drivers manually with deep-linked instructions** | Current v3.0 behavior. Onboarding hits 90-120s on cold install per `install_vm_matrix.json`. Friction kills virality. **Recommendation: REJECT** — fails the v3.1 "one-click hard requirement" memory `project_one_click_install_hard_req` | ❌ Rejected (status quo) |

#### NEW components

| Component | Path | Role | Why it lives here |
|-----------|------|------|-------------------|
| `installer/companion/fetch_drivers.sh` (Mac) | NEW file | Downloads BlackHole 0.6.x signed pkg from `existential.audio` releases over HTTPS, verifies SHA-256 against pinned manifest, runs `installer -pkg ... -target /` with sudo prompt | Sibling to existing `installer/windows/vibemix-installer.iss` — installer-related artifacts live at `installer/<platform>/` |
| `installer/companion/fetch_drivers.ps1` (Win) | NEW file | Downloads VB-CABLE signed installer from `vb-audio.com`, verifies SHA-256, runs with elevated UAC | Same sibling pattern |
| `installer/companion/driver_manifest.json` | NEW file | SHA-256 + canonical URL + version + license-acknowledgment text for BlackHole + VB-CABLE. Read by both `.sh` and `.ps1` so version bumps land as a one-line edit | Single source of truth; mirrors `scripts/dist/install_vm_matrix.json` pattern |
| `installer/companion/audio_config.py` | NEW file | Post-driver-install: programmatically configure macOS Multi-Output Device (via AppleScript or `SwitchAudioSource`) + Windows default playback device routing. Runs inside the sidecar wizard slot | Single Python module callable from existing `install.blackhole_probe` flow |
| `scripts/audit/check_companion_signing.sh` | NEW file | CI gate verifying companion scripts are signed (codesign on Mac, Authenticode on Win) — runs in `release.yml` SIGN stage after the main app sign | Mirrors existing `scripts/dist/verify_signed.py` discipline |

### Feature 2 — Dep Audit / Pin Surface

**Goal:** Every runtime dep (Python + Rust + Node) pinned with rationale, license, and install-impact rating (green/yellow/red). Cull unused/stale deps. Single source of truth so future bumps land deliberately.

#### Architecture decision: generated AUDIT.md from existing lockfiles

v3.0 already has the lockfile substrate — `uv.lock` (Python, 2065 lines), `tauri/src-tauri/Cargo.lock`, `tauri/ui/package-lock.json`. v3.0 also has the CVE infrastructure — `python-cve.yml` (pip-audit + osv-scanner), `rust-cve.yml` (cargo-audit + cargo-deny), `sbom.yml` (syft SPDX). What's MISSING is the rationale + install-impact column.

The audit lives as **generated documentation**, not a new lockfile. Inputs are existing lockfiles; output is a single committed markdown table. CI gate compares freshness (last-generated timestamp vs. lockfile mtime) and fails if the audit is stale.

#### NEW components

| Component | Path | Role |
|-----------|------|------|
| `docs/AUDIT.md` | NEW file (generated) | The dep-audit surface itself. Three tables: Python (from `uv.lock`), Rust (from `Cargo.lock`), Node (from `package-lock.json`). Each row: name, pinned version, source (direct/transitive), license, install-impact (green/yellow/red), rationale (1 sentence). Committed to repo. Updated by the generator. |
| `scripts/audit/dep_audit.py` | NEW file | The generator. Reads three lockfiles, joins to a hand-maintained `scripts/audit/dep_ratings.json` for the rating + rationale columns, emits `docs/AUDIT.md`. Idempotent on identical inputs. |
| `scripts/audit/dep_ratings.json` | NEW file | Hand-maintained mapping `{dep_name → {install_impact: "green"|"yellow"|"red", rationale: "<= 120 chars", license_override?: "..."}}`. Source of truth for the human judgment column. Reviewed at every dep bump |
| `scripts/audit/check_audit_freshness.sh` | NEW file | CI gate. Reads `docs/AUDIT.md` last-generated timestamp + `uv.lock` / `Cargo.lock` / `package-lock.json` mtimes; fails if any lockfile is newer than AUDIT.md. Mirrors `check_v5_fonts.sh` / `check_v5_copy.sh` discipline |
| `.github/workflows/dep-audit.yml` | NEW workflow | Runs `dep_audit.py` + `check_audit_freshness.sh` on PRs touching `uv.lock` / `Cargo.lock` / `package-lock.json` / `pyproject.toml` / `tauri/src-tauri/Cargo.toml` / `tauri/ui/package.json` |

#### MODIFIED v3.0 components

| Component | Path | Rationale |
|-----------|------|-----------|
| `.github/workflows/python-cve.yml` | EXISTING | Add post-step: if pip-audit flags a CVE on a `green`-rated dep, downgrade rating to `red` automatically in a PR comment annotation. Does NOT auto-edit `dep_ratings.json` — that's a human call |
| `.github/workflows/rust-cve.yml` | EXISTING | Same pattern for cargo-deny |

### Feature 3 — E2E MacBook Test Harness

**Goal:** Kaan runs the SHIPPED `.dmg` on his MacBook with a real DJ set, validates functional + visual + aesthetic + usability. Release-blocking issues surface as gap-closure items.

#### Architecture decision: `tests/e2e/macbook/` sibling to existing seam tests; driven by `scripts/e2e/run_macbook_pass.sh`

The existing `tests/e2e/` directory holds **in-process seam tests** (P18→P20, P19→agent, P25→P28, P27→eval_gate, P31→ws_bus). These verify integration but NOT the SHIPPED ARTIFACT. v3.1's e2e is fundamentally different — it runs the signed DMG on real hardware.

Architecturally, this is **two harnesses with shared fixtures**:
- **Automated portion** (`scripts/e2e/run_macbook_pass.sh` + `tests/e2e/macbook/*.py`): orchestrates DMG install + sidecar launch + IPC message capture + screenshot grab + audio capture. Pytest-driven where possible.
- **Manual portion** (`docs/e2e/MACBOOK-PASS-PROTOCOL.md`): the steps Kaan walks through during a real DJ set. Mirrors `eval/EAR-TEST-PROTOCOL.md` pattern.

#### NEW components

| Component | Path | Role |
|-----------|------|------|
| `tests/e2e/macbook/__init__.py` + `conftest.py` | NEW | Shared fixtures: DMG path discovery, sidecar process spawn helper, IPC client adapter, screenshot grab via `screencapture -R <rect>` |

#### Asset storage decision: in-repo, NOT git LFS

- **Total size after swap:** 5 × ~800KB (target midpoint) = ~4 MB added to the existing ~21.67 MB mascot bundle → **~25.67 MB total**.
- **Tier 1 ceiling:** 25 MB. **This will fail by ~0.67 MB.**

Two options:
- **(A) Hold the 25 MB ceiling** — re-tune draco compression per-clip to fit. `scripts/glb_optimize.py` already shells out to draco. Tighter compression (level 8 instead of 7) typically nets 15-20% reduction. **Recommendation: PREFER** — preserves the bundle discipline established in v2.1

Git LFS is **rejected** for these files:
- 5 files at ~800 KB each = ~4 MB total. LFS overhead (clone bandwidth, CI checkout cost, contributor setup friction) outweighs the storage cost.
- Repo is OSS-targeted at DJ users for a 500-1000-star ambition (per memory `project_github_star_goal`); LFS dependency adds first-clone friction
- LFS is reserved for the v3.0 carryover `EVAL-CORPUS-WAVS` (6 × 30-min DJ session WAVs ≈ 200 MB) — that's the right size threshold
