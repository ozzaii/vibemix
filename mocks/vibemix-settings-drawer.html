THE DRAWER
SLIDES OVER
— NEVER OVER THE SET —

The session is the deck. The drawer is a second pane of glass that slides in from the right, lays a faint scrim over the live UI, and holds the meta-tasks: recordings, retention, performance. When closed it's gone — when open, the set is still readable behind it.

Built after the 2026-05-14 critique found the shipped components had drifted (no design mock = visual entropy). This file is the missing target — one drawer surface, one breathing border, every row earns its space, amber appears only on what's active or destructive.

§ 05

Anti-Patterns Compliance

what was deliberately avoided
/* settings-drawer · anti-pattern register */ - side-stripe accent borders on rows + background-tint hover only · the row reads as flat panel - gradient text (amber→pale on the readout) + solid silk on glass-3 recess · text-shadow does the depth - hero-metric template ("12 SESSIONS / —— total recordings") + inline group-head meta · "12 sessions · 480 MB" silkscreen - modal-first delete confirmation + optimistic delete + 4s undo toast · see §06 - glassmorphism stacking (group cards on top of drawer glass) + ONE glass surface (the drawer) · groups are flat dividers - 3+ concurrent border-anim sweeps (anxiety, not life) + ONE breathing border · the drawer · 26s sweep - AI-assistant tropes (sparkles, "✨ smart cleanup") + humanized readable line · "auto-pruning after 7 days" - decorative icons next to every label + icons only on action cells (play / reveal / delete) // every element earns its space. Pioneer-grade restraint: // if it doesn't change state or carry data, it doesn't ship.

interaction script
  1. User clicks × on a row.
  2. Row vanishes immediately — no confirm dialog.
  3. Toast anchors bottom-right of the drawer.
  4. Saira 11px --silk-65 body, amber underline on undo?.
  5. Progress hairline counts down 4 seconds along the bottom edge.
  6. Clicking undo? restores the row + dismisses toast.
  7. After 4s: toast fades, delete is committed to disk via ipc.recordings.delete.
Why no modal? Because the user already consented by clicking the trash icon. A modal would add friction without adding safety — the undo toast adds safety without adding friction. Critique 2026-05-14 flagged the prior confirm-dialog pattern as modal-first; this is the agreed replacement.
